# UX Guidelines PracticeRaptor

## 1. Введение

Этот документ описывает дизайн пользовательского опыта (UX) для платформы PracticeRaptor. Наша главная цель — создать **единообразный и консистентный пользовательский опыт** на всех платформах (CLI, Telegram Bot, Web), обеспечивая **аналогичность функционала**. Пользователь должен иметь возможность выполнять одни и те же задачи и получать схожую обратную связь, независимо от выбранного интерфейса.

## 2. Основной Пользователь

В центре нашего внимания — **"Ученик"**. Это разработчик, который стремится улучшить свои навыки решения алгоритмических задач. Он ценит эффективность, ясность, мгновенную обратную связь и возможность использовать платформу как в командной строке, так и через более привычные каналы (веб, мессенджер), ожидая при этом одинакового уровня функциональности и логики.

## 3. Ключевые Принципы UX-Консистентности

1.  **Функциональная Аналогия:** Каждое основное действие пользователя и соответствующая ему информация должны иметь прямой и понятный эквивалент во всех интерфейсах. Например, выбор задачи с фильтрами через команду CLI, команду в Telegram или комбинацию фильтров и кнопку на веб-сайте — все они должны приводить к выбору одной и той же задачи и вызову одного и того же ядра.
2.  **Единообразная Обратная Связь:** Статус выполнения кода (Accepted, Wrong Answer, Error, Timeout), системные сообщения (загрузка, успех, ошибка) должны отображаться схожим образом, чтобы пользователь интуитивно понимал, что происходит.
3.  **Унифицированный Доступ к Информации:** Структура и навигация по основным разделам (каталог задач, профиль, настройки, решение задачи) должны быть логически связаны и предсказуемы на всех платформах.
4.  **Следование Ядру:** Все действия пользователя должны инициировать вызов соответствующих функций ядра PracticeRaptor, обеспечивая единую бизнес-логику.

## 4. Информационная Архитектура и Навигация

Мы структурируем информацию и функционал приложения таким образом, чтобы пользователь мог легко находить необходимое.

*   **Основные разделы:** Каталог задач, Детали задачи, Решение задачи, Профиль пользователя (Прогресс, Статистика, Настройки), Подсказки/Решения.
*   **Навигация:** Подробное описание того, как осуществляется переход между этими разделами в каждом интерфейсе, сохраняя логическую связь.

(См. `ux/information_architecture/` для деталей)

## 5. Пользовательские Потоки и UX-Паттерны

Детально проработаны основные сценарии взаимодействия пользователя с платформой, с акцентом на аналогичность действий и реакций системы.

*   **Основные Пользовательские Сценарии (User Journeys):**
    *   Выбор задачи (с фильтрами, случайный)
    *   Просмотр деталей задачи
    *   Решение задачи (ввод кода, проверка, отмена)
    *   Просмотр прогресса и статистики
*   **Ключевые UX-Паттерны:**
    *   Модели обратной связи и обработки ошибок
    *   Представление данных (списки, детали, статистика)
    *   Модели взаимодействия (выбор, подтверждение, отмена)

(См. `ux/user_journeys/` и `ux/patterns/` для деталей)

## 6. Детализация UX для Интерфейсов

Общие принципы и паттерны применяются к каждому интерфейсу с учетом его особенностей, но с сохранением максимальной аналогии.

*   **CLI (Stage 1.6):** Детальное описание команд, аргументов, вывода, цветового оформления и интерактивных элементов, реализующих UX-принципы.
*   **Telegram Bot (Stage 2):** Адаптация UX-паттернов для команд, inline-кнопок, форматирования сообщений.
*   **Web (Stage 3):** Проработка UX с учетом визуальных элементов, форм и навигации браузерной среды.

(См. `ux/interfaces/` для деталей)

## 7. Приоритеты и Реализация

На этапе **Stage 1.6** основное внимание уделяется детальной проработке UX для **CLI**, максимально приближая его к общему видению и закладывая основу для аналогичной реализации в других интерфейсах.
